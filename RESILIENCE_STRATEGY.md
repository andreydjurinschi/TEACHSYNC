# TeachSync: отказоустойчивость микросервисов

## Что уже реализовано

В системе используется несколько уровней отказоустойчивости:

1. **Изоляция данных по сервисам**  
   Каждый бизнес-сервис имеет собственную базу данных. Отказ одного сервиса не уничтожает данные другого сервиса.

2. **Асинхронные события через Kafka**  
   Уведомления, активность пользователей и часть аналитики строятся на событиях. Если сервис-потребитель временно недоступен, событие остается в Kafka и может быть обработано после восстановления.

3. **Real-time доставка через SSE с сохранением в БД**  
   Уведомление сначала сохраняется в `notification-service`, а затем отправляется активному пользователю через SSE. Если пользователь не подключен или соединение оборвалось, уведомление не теряется и будет доступно после обновления страницы.

4. **Timeout + Circuit Breaker для Feign**  
   Для межсервисных HTTP-вызовов добавлены короткие таймауты. Это защищает сервисы от долгого зависания, если соседний сервис перестал отвечать.

5. **Локальный TTL-кэш справочных данных**  
   Несколько сервисов сохраняют последние успешно полученные справочные данные от соседних сервисов:
   - данные преподавателей;
   - данные групповых курсов;
   - данные расписаний для отображения;
   - размер группы для расписания;
   - список курсов преподавателя для профиля пользователя.

   Если соседний сервис временно недоступен, сервис пытается показать последнюю успешную версию данных из кэша.

6. **Системные уведомления администраторам**  
   Если операция не может быть безопасно выполнена по кэшу, сервис публикует `SYSTEM_ALERT` в Kafka. `notification-service` превращает это событие в уведомление для администраторов.

## Почему кэш используется не везде

Кэш нельзя использовать для всех операций подряд. В TeachSync он применяется только там, где допустима временно устаревшая информация:

- вывод расписания;
- карточки замен;
- статистика;
- профиль пользователя;
- отображение имени преподавателя или курса;
- отображение курсов преподавателя в профиле;
- выдача group-course данных для расписания.

Для критичных команд система продолжает требовать актуальный ответ от сервиса-источника:

- создание расписания;
- изменение расписания;
- подтверждение замены;
- проверка доступности преподавателя;
- проверка конфликтов по времени;
- проверка специализации преподавателя.

Это важно: если использовать устаревший кэш в командах, система может принять неправильное бизнес-решение, например назначить преподавателя, который уже занят.
В таких случаях сервис не подменяет данные кэшем, а отправляет администратору системное уведомление о проблеме межсервисного взаимодействия.

## Поведение при отказе

Пример: временно недоступен `users-service`.

- `schedule-service` может продолжить показывать расписание с последними известными именами преподавателей.
- `replacement-service` может продолжить показывать список заявок на замену с последними известными данными.
- `course-service` может продолжить отдавать group-course данные с последними известными именами преподавателей.
- `users-service` может показать последние известные курсы преподавателя в профиле.
- Команды, где нужно точно проверить преподавателя, могут быть временно отклонены до восстановления `users-service`.

Такой режим называется **graceful degradation**: система не падает полностью, а сохраняет доступными безопасные функции.

## Что можно добавить дальше

Для промышленной версии можно усилить отказоустойчивость:

- Redis вместо локального in-memory кэша, чтобы кэш был общим для нескольких экземпляров сервиса;
- health-check endpoints через Spring Actuator;
- retry с backoff для временных сетевых ошибок;
- outbox pattern для гарантированной публикации Kafka-событий после записи в БД;
- docker `healthcheck` и `restart: unless-stopped` для всех сервисов;
- метрики Prometheus/Grafana по ошибкам Feign, Kafka lag и количеству SSE-подключений.
